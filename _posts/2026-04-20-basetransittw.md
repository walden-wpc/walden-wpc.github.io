## 這是什麼？爲什麼要做？

一開始出發點其實很單純，我想在學校或公司直達的公車路線沿線找房子。 而現有工具要嘛太複雜，要不就是得一站一站自己查。後來才發現這個問題看似簡單，目前市面上卻沒有一個工具能直接回答，Google Maps 可以查路線，但無法從某個站出發、同時探索所有可達的範圍；政府資料開放平台有原始數據，但視覺化呈現並不夠直觀好用。

BaseTransitTW 就是為了填補這個空缺而做的。這個工具的核心是**等時線**：在地圖上畫出從指定起站出發、在設定時間內能搭公車到達的所有範圍。除了等時線，我也想直接解答另一個問題：「有沒有被忽略的交通節點」，所以也做了**發車密度**視覺化，讓每條路線的班距、每個公車站的公車密度可以直接反映在顏色上，同時與其他資訊顏色做區別，讓人能夠一眼在同一個地圖上比對並獲取不同資訊。

## 設計決策

在此之前我沒有前端開發背景，因此寫程式的部分我大量使用了 AI 工具協助，絕大部分程式碼是透過 Vibe coding 建起來的。但我刻意讓整體架構保持清晰、模組切分明確、註解寫得夠詳細。這樣萬一 token 爆了或某個地方要手動維修，還能看得懂自己的系統在幹嘛。這應該是 2026 年把好點子變成穩定產品最合理的做法。

中間花了不少時間在改善UI上，由於開發圖中新增了很多顯示方式與功能，爲了避免看起來很繁雜，也希望使用起來能順手一點，我最後採用的方案主軸是**按使用頻率分層**：通勤圈切換、站牌搜尋永遠在頂部（每次查詢都會用）、發車密度放在顯眼位置作為主打功能、真正消耗算力的進階選項則收折藏起並附上警告。這樣就可以在維持簡潔的同時，保留很多彈性能針對細節做調整。

必須特別提及，發車密度是我在這個 App 裡最重要的功能之一。班距是一個對通勤品質影響巨大、但幾乎從來不被直接視覺化的數字。目前開啓此功能後，路線線條的顏色從綠到紅直接對應班距從密到疏，也可以看出哪個公車站是重要的轉乘節點，完全解決了我的痛點。

## 技術選擇

- Dijkstra 
- Next.js
- MapLibre
- Turf.js
- Binary Min-Heap
- TDX API

核心設計原則只有一條：零成本運作。因爲我暫時不打算付伺服器費用：）

這條原則幾乎決定了一切：沒有後端伺服器。所有路徑計算都在你的瀏覽器裡跑，不存在任何雲端運算，這樣除了省錢之外安全性還很高，因爲線上根本沒有方法可以記錄你查了什麼。

![介面展示](/assets/images/basetransit_mockup_1.png)

等時線計算使用 **Dijkstra 演算法** ，並包含候／轉車時間預估，等車的時間也有算進去。也因此跑起來很吃資源，放在主執行緒會讓頁面凍結，所以我把它丟進 Web Worker 獨立處理。計算途中地圖還是可以操作，而且會隨著每五分鐘時間帶逐步更新，不用等全部跑完才看到畫面。

Dijkstra 用來跑「等時線擴散」，概念是從起點出發，沿著公車路網往外擴，記錄到每個站牌的最短時間，直到超過時間上限為止。
邊的權重不是單純的行車時間，有加上候車懲罰，暫時定義爲班距除以二。舉例來說，一條十分鐘一班的路線，平均要等五分鐘，這五分鐘會被計入總花費時間。
轉乘的處理用平面矩陣 $dist[站牌][轉乘次數]$ 分開追蹤。舉例來說，同一個站牌可能有「轉乘零次、花了二十分鐘」和「轉乘一次、花了十五分鐘」兩種抵達方式，兩個都保留，不會互相覆蓋。

Binary Min-Heap 是判斷優先路徑的邏輯，每次從待處理的站牌裡取出「目前花費時間最少的目標」這個操作可以在 $O(log N)$ 完成，整體演算法複雜度是 $O((V+E)logV)$。如果用未排序矩陣的話，取最小值要掃過全部元素，V 個站牌全部處理完的複雜度是 $O(V²)$。差距顯而易見。

Web Worker 部分，因為跑全台路網時 Dijkstra 可能要好幾秒，如果放在主執行緒高機率會讓整個頁面凍結、無法互動。用 Web Worker 讓它在背景獨立執行，每算完一個五分鐘時間帶就回傳一次進度，地圖上的等時線範圍會逐步擴大，不用等全部跑完才看到結果。

資料的部分，用 Python 腳本從交通部 TDX 平台抓取，預處理成輕量的 Graph JSON。靜態產物丟上 GitHub Pages。

路網圖是這個 App 的核心。全台22縣市的站牌與路線資料來自交通部 TDX 開放資料平台，由 Python 腳本定期抓取後，轉換成以站牌為節點、乘車時間為邊的圖資 JSON，再部署為靜態檔案。**整個 App 沒有後端**：計算在使用者的裝置上跑，資料從 CDN 拉，不需要伺服器。
而跨縣市／通勤圈／公路客運路網的挑戰在於銜接，兩個鄰近的站牌在資料上互相不認識，但現實中步行一段就能轉乘。因此我在圖資合併時有對跨城市的鄰近站牌動態建立步行邊，讓等時線能自然地跨過縣市邊界延伸。

### 目前涵蓋範圍


全台22縣市公車路線，包含國道客運。各縣市資料由自動化管線維護，並有獨立的QC工具可以管理、標記後端資料品質及狀態，確保地圖上顯示的路線正確且走向盡量貼近實際。

## 總結

對我來說，BaseTransitTW 最重要的意義是：我確實擁有處理真實世界複雜資料、並將其互動視覺化的能力。
本質上來說這些邊與節點其實和我的碩論非常相似，只是主體換成了公車路網，這當中有很多相似可通用之處，因此在框架理解上並不困難，主要的時間都花在UI設計、演算法邏輯、要用什麼方法渲染畫面、效能取捨等等。

專案仍在維護、資料不定期更新中。對公共運輸資料、前端效能或純前端地理運算有興趣的話，歡迎直接翻原始碼。
